## Strategies
- **PickerCloneStrategy projects to a 3-field price-only subspace at runtime** — the committed profile uses all seven factor fields, but at signal time only `log_market_cap` (proxied by log dollar volume), `momentum_12_1`, and `realised_vol_60d` are computable from OHLCV. We compare on the subspace, which throws away the value/quality/leverage signal the profile actually encodes. - **PickerCloneStrategy equal-weights the basket** — real picker portfolios value-weight by conviction (Berkshire is 40%+ AAPL). The clone gives every top-N name 1/N weight. ## Backtest
- **`buy_and_hold` ignores entry timing of late-listed tickers** — current impl forward-fills NaNs and assumes every ticker starts at the universe's first bar. If a ticker IPOs mid-window we currently buy phantom shares at its first available price; better to defer that ticker's allocation until its first non-NaN bar and rebalance the remainder. — S — `backtest/benchmarks.py::buy_and_hold`.
- **`random_entry_monte_carlo` uses i.i.d. Bernoulli per (path, bar, ticker)** — this matches *average* exposure but not the *autocorrelation* a real strategy exhibits. Add a block-Bernoulli variant (run lengths drawn from a geometric distribution) so the null distribution's realised vol is closer to a real strategy's. — M — `backtest/benchmarks.py::random_entry_monte_carlo`.
- **`random_entry_monte_carlo` Sharpe uses fixed 252 annualisation** — fine for daily bars, wrong for intraday. Rigorous annualisation lands in Task 7b's `stats.py`; remove the duplicate here once it does. — S — `backtest/benchmarks.py`.
- **`buy_and_hold_spy` fetches via the data layer on a miss** — but does so synchronously inside a request. For the FastAPI `/benchmarks` path this can block for several seconds on a cold SPY backfill. Either pre-warm SPY at startup or move the fetch to a background task. — S — `backtest/benchmarks.py::buy_and_hold_spy`, `api/app.py`.
- **Drawdown thresholds are hand-picked** — `-5% / -15%` for calm/mild/severe is conservative but uncalibrated. After Task 7c walk-forward lands, fit thresholds to historically known regime breaks (2008, 2020, 2022) and document the fit. — M — `backtest/regimes.py`.
- **Volatility terciles are window-relative, not cross-time-comparable** — a low-vol bar in 2020's input window is not a low-vol bar in 2017's. For cross-period comparisons add an optional `quantiles=` arg using fixed historical quantiles. — S — `backtest/regimes.py::tag_volatility`.
- **`regime_split.exposure` uses a non-zero-return proxy** — true position-time exposure requires the vectorbt portfolio (`asset_value() != 0`). Add an overload that accepts the BacktestResult directly so the regime split can read the real exposure. — S — `backtest/regime_split.py`.
- **No `/regimes` endpoint yet** — the regime tags exist server-side but the UI (Task 7d) will have to load + tag client-side or via the existing data endpoints. Expose `POST /regimes` that returns the tag DataFrame for a window. — S — `api/app.py`.
- **VIX backfill is not auto-triggered** — `buy_and_hold_spy` falls back to YFinance for SPY but `regimes.tag_volatility` callers must backfill `^VIX` themselves. Add a `_ensure_series_in_clickhouse` helper. — S — `data/backfill.py`.
- **`spa_test` ships White's Reality Check, not Hansen SPA** — full Hansen (2005) re-centres the bootstrap distribution at zero for any competitor with negative sample mean, which makes the test strictly more powerful when many competitors are obviously bad. We ship White (2000) — the Reality Check ancestor — with re-centring at the sample mean only (not the Hansen v_l / v_c / v_u three-statistic variant). For typical sweep grids where competitors are roughly comparable the difference is small; for grids with many "noise" competitors SPA would reject more often. — M — `backtest/stats.py::spa_test`.
- **No HAC standard errors on Sharpe** — the rigorous parametric CI for the Sharpe ratio uses Newey-West / HAC variance-of-Sharpe (Lo 2002, "The Statistics of Sharpe Ratios") rather than a bootstrap. We ship only the stationary bootstrap; for callers who want a closed-form CI under serial correlation, HAC would be cheaper and more transparent. — M — `backtest/stats.py`.
- **Bootstrap CI is percentile, not BCa** — bias-corrected & accelerated bootstrap (Efron 1987) is closer to nominal coverage for Sharpe at small samples (the percentile method has a known undercoverage gap, which is why `test_sharpe_ci_contains_true_value_most_of_the_time` runs at α=0.01 to get ≥90% coverage in 50 trials). Add a `method="bca"` option. — M — `backtest/stats.py::bootstrap_ci`.
- **PSR uses Bailey's raw 4th-moment kurtosis, not excess** — implemented as `scipy.stats.kurtosis(fisher=False, bias=True)` per the paper. This is correct but easy to get wrong by callers eyeballing the formula in a textbook. A second optional argument `kurtosis_convention='raw'|'excess'` with a deprecation path would help. — S — `backtest/stats.py::probabilistic_sharpe_ratio`.
- **DSR `_expected_max_iid_normal` ignores correlation between trials** — Bailey & López de Prado (2014) treat sweep trials as independent normals. In a real grid sweep adjacent params produce highly correlated returns; the effective `n_trials` is smaller than `len(grid)`, so DSR over-deflates. Add an "effective N" estimator from the correlation matrix of trial returns. — M — `backtest/stats.py::deflated_sharpe_ratio_from_sweep`.
- **`/stats` recomputes three independent bootstrap batches** — Sharpe / max-DD / total-return each loop over fresh resamples. A single resample-once-then-apply-three-metrics path would be ~3× faster and is the standard pattern in `arch` / `scikit-learn`. — S — `api/services.py::run_stats`.
- **Walk-forward has no purged k-fold (López de Prado AFML §7.4)** — `walk_forward` does a chronological split per fold, but with overlapping `step < test_size` we do not purge the bars that leak information across the train/test boundary. The proper fix is to drop a `purge_bars` buffer either side of the split (≥ the strategy's lookback) and, for combinatorial purged CV, embargo a tail of train bars after each test slice. — M — `backtest/walkforward.py::walk_forward`.
- **Walk-forward "best combo" picker is greedy Sharpe-max** — picks the highest-Sharpe combo on the train fold with no PSR / DSR penalty for the size of the grid. A real production picker should deflate the train Sharpe by the number of trials (DSR — already in `stats.py::deflated_sharpe_ratio_from_sweep`) before promoting. — S — `backtest/walkforward.py::_best_combo`.
- **`decay_slope` uses plain OLS** — one outlier fold (e.g. a vbt NaN on a degenerate window) can swing the slope. Robust regression (Theil-Sen) or a paired bootstrap CI on the slope would be more defensible as a diagnostic. — S — `backtest/walkforward.py::_ols_slope`.
- **`_portfolio_returns` averages tickers equal-weight** — matches `summarize`'s cross-ticker mean, but a real multi-ticker portfolio Sharpe should use vbt's portfolio-level value, not a per-ticker average. Once `BacktestResult` exposes a single composite equity curve this collapses to a one-liner. — S — `backtest/walkforward.py::_portfolio_returns`.
- **No Newey-West / HAC standard errors on alpha t-stat** — `market_attribution` uses plain-OLS residual variance for the SE. Returns are typically autocorrelated at lag 1 (volatility clustering), so the SE is too tight and the t-stat is over-confident. The standard fix is Newey-West with `floor(4 * (n/100)^(2/9))` lags. — M — `backtest/attribution.py::market_attribution`.
- **No multi-factor attribution (Fama-French 3F / 5F / Carhart 4F)** — `market_attribution` is single-factor (CAPM). For a multi-factor decomposition the same OLS machinery extends trivially, but the data layer doesn't have FF factors loaded. Pair a `data.backfill_ff_factors` helper with a `multi_factor_attribution` function. — M — `backtest/attribution.py`.
- **Child-signal attribution attributes return, not variance** — contributions sum to the combined return but say nothing about each signal's contribution to portfolio volatility. A risk-decomposed variant (`Σ_i w_i² σ_i² + 2 Σ_{i<j} w_i w_j σ_ij`) would tell the user which signal drove the realised risk. — M — `backtest/attribution.py::child_signal_attribution`.
- **`/walkforward` runs the sweep + per-fold backtest synchronously inside the request** — a 10-fold walk-forward on a 3-strategy × 16-combo grid is several seconds, which exceeds typical UI tolerances. A `POST /walkforward/start` → `GET /walkforward/{job_id}` async pattern (or SSE streaming per fold) is the proper fix. — M — `api/app.py::walkforward_endpoint`.
- **`fit_combined_weights_walk_forward` runs each child as its own backtest per fold** — N children × M folds round-trips through `run_backtest`. For the default 6-child line-up × ~5 folds × ~6y daily bars this is the dominant cost of weight learning. Memoise per (child, train_slice_hash) so a re-fit on the same window is instant. — M — `backtest/walkforward.py::fit_combined_weights_walk_forward`.
- **`explain_trades` ignores trade direction** — vbt's records carry a `direction` column for long-short portfolios; we hard-code `long_entry` / `long_exit` because the strategy is long-only. When `Signals` lifts to long-short, read the direction column and emit `short_entry` / `short_exit` accordingly. — S — `backtest/explainability.py::explain_trades`.
